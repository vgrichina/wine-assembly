Updated Findings
----------------
1. FS segment handling is no longer the top open bug.
   - src/07-decoder.wat now applies segment overrides centrally after decode_modrm.
   - apply_seg_override is idempotent, so older emitter-side calls are safe guards.
   - test/test-fs-prefix.js covers FS TEST/CMP/MOVZX/MOVSX/CMPXCHG/XADD and baseline MOV.
   - Remaining risk: coverage is focused, not exhaustive for every memory opcode form.

2. Address-size override 0x67 is now a deliberate fail-fast gap.
   - src/07-decoder.wat still parses prefix_67.
   - It now traps explicitly when prefix_67 is set instead of silently decoding 16-bit
     addressing as 32-bit ModRM/SIB.
   - Compatibility impact remains: any app using 0x67 address-size override will stop.
   - This is better than silent corruption, but real 16-bit addressing is still missing.

3. GS segment override is intentionally unsupported.
   - src/10-helpers.wat seg_adj traps GS.
   - src/07-decoder.wat apply_seg_override traps GS.
   - FS is the Win32-relevant path for this emulator, but GS-prefixed code will stop.

4. FPU is intentionally x87-lite and has known semantic holes.
   - src/06-fpu.wat documents f64 instead of 80-bit extended precision.
   - Rounding/exceptions/status are partial.
   - FXTRACT remains placeholder stack-effect-only behavior.
   - Unknown x87 encodings crash through fpu_crash_op.

5. SEH/C++ EH is incomplete.
   - src/11-seh.wat handles simplified SEH patterns.
   - _CxxThrowException, __CxxFrameHandler, and _global_unwind2 still crash in
     src/09a-handlers.wat.
   - Apps using real C++ exception unwinding will fail outside the narrow handled cases.

6. Direct3D Immediate Mode remains a compatibility scaffold.
   - src/09aa-handlers-d3dim.wat started as generated stubs.
   - src/09ab-handlers-d3dim-core.wat still describes Phase 0/Phase 1+ helper scaffolding.
   - Unsupported execute-buffer ops crash.
   - Unsupported draw cases crash.
   - Texture, material, lighting, viewport, callback, and full fixed-function behavior are incomplete.

7. DirectX wrappers mostly keep apps moving, not fully emulate devices.
   - DirectDraw has many no-op or shallow methods: clipper, surfaces, attached surfaces,
     clip lists, VBlank, cooperative level.
   - DirectSound has shallow buffer/format/lock/seek behavior.
   - DirectInput returns limited state; mouse deltas are explicitly TODO.

8. Controls continue to grow around combo/listbox behavior.
   - src/09c3-controls.wat is now 6020 lines.
   - Recent changes route combo popup mouse events to the inner listbox and post combo
     notifications to avoid nested dialog reentrancy.
   - The explicit fail-fast control crash remains around the feedback/memory stub path.
   - Controls are materially better than pure stubs, but still not full USER32/COMCTL32.

9. Win32 API coverage is wide but shallow.
   - Major hard-stub clusters: wide-char APIs, GDI mapping/fonts/metafiles/printing,
     clipboard, menu query/mutation, MDI/dialog helpers, process/pipe/file metadata,
     registry, atom/profile APIs.
   - Many soft stubs return plausible success without backing state.

10. Debug/app-specific tracing may still exist in instruction/export surface.
    - Earlier call/push debug traces in src/05-alu.wat appear reduced by line-count changes,
      but this audit did not prove all debug-only paths are gone.
    - src/13-exports.wat still exposes host/test/debug-style helper exports.
    - Risk is trace noise and app-specific behavior leaking into normal runs.

# Code Review Report: Wine-Assembly x86 Emulator

## 1. Summary
This report details the current state of the Wine-Assembly x86 emulator, focusing on API gaps, misimplementations, and the identification of "silent" stubs. A critical finding is that many Win32 APIs are implemented as minimal return-0/1 stubs, which hides implementation gaps and makes debugging guest applications difficult.

## 2. API Gaps & Silent Stubs (src/parts/09a-handlers.wat)
A systemic review of the API handlers identified over 100 functions that fail silently.

### 2.1 Critical Unicode (W) Variants
Most Unicode variants are currently non-functional stubs. Applications attempting to use NT-style Unicode APIs will fail immediately or behave unpredictably:
*   **Windowing:** `RegisterClassW`, `RegisterClassExW`, `CreateWindowExW`, `SetWindowTextW`, `GetWindowTextW`.
*   **Messaging:** `SendMessageW`, `PostMessageW`.
*   **Resources/Libraries:** `LoadLibraryW`, `LoadCursorW`, `LoadIconW`, `LoadMenuW`.
*   **String/Utility:** `wsprintfW`, `wvsprintfW`, `lstrcatW`.

### 2.2 System & Environment Stubs
Several core system APIs return hardcoded "success" or "empty" values without performing any action:
*   **Time:** `GetLocalTime` (returns zeroed structure), `GetTimeFormatA`, `GetDateFormatA`.
*   **Profile/Registry:** `GetProfileStringA`, `GetProfileIntA`, `WriteProfileStringA`.
*   **Locales:** `GetLocaleInfoA`, `GetUserDefaultLCID`, `GetACP`, `GetOEMCP`.
*   **Filesystem (Wide):** `FindFirstFileW`, `GetFileAttributesW`, `GetCurrentDirectoryW`, `DeleteFileW`.
*   **Critical Sections:** `InitializeCriticalSection`, `EnterCriticalSection`, `LeaveCriticalSection` (implemented as no-ops).

### 2.3 Misimplementations
*   **`MultiByteToWideChar` / `WideCharToMultiByte`**: Currently performs a direct 1:1 byte copy. It does not respect code pages or handle multi-byte sequences correctly.
*   **`GetProcAddress`**: Leaks guest heap memory by creating a new thunk on every call, even for the same function. It also lacks support for ordinal lookups.
*   **`GetModuleHandleA`**: Hardcoded to return the `image_base`, which breaks for applications expecting to load or query multiple DLLs.

## 3. Host Import Gaps (lib/host-imports.js)
*   **`read_file`**: Currently a hardcoded stub returning 0. File I/O is not yet bridged to the host filesystem.
*   **GDI Text**: `gdi_text_out` uses a fixed 13px monospace font regardless of the font selected into the DC.
*   **Window Styles**: Many `dwExStyle` and `dwStyle` flags are ignored in `create_window`.

## 4. Policy: Transition to "Crash on Run"
To improve debuggability, the project is moving away from silent stubs.
*   **Mechanism**: All stubs should invoke `(call $crash_unimplemented (local.get $name_ptr))`.
*   **Host Behavior**: The host implementation in `host-imports.js` logs the EIP, ESP, and the name of the missing API, then halts execution.
*   **Exception**: Minimal functional stubs like `GetLastError` (returning a global error code) and `GetModuleHandleA` (returning the main base) should be retained until full implementations are ready, as they are often required for basic application startup.

## 5. Confirmed Critical CPU Bugs
*   **Decoder:** `0x8F` (POP [mem]) uses stale ESP; `SETcc` clobbers registers when targeting memory.
*   **ALU:** `ADC`/`SBB` destroys `flag_res` on carry, corrupting subsequent flag checks.
*   **PE Loader:** Fails to zero the BSS area, leading to non-deterministic behavior in global variables.
